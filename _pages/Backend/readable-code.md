## 코드는 실행보다 오래 남는다

기능을 끝내는 순간에는 코드가 잘 동작하는지만 보이기 쉽다. 하지만 시간이 지나 다시 그 코드를 열어보면, 진짜 비용은 실행 여부보다 이해 가능성에서 나온다. 좋은 코드는 다음 사람이 빠르게 고칠 수 있게 남아 있어야 한다.

읽기 쉬운 코드를 만들 때 가장 먼저 볼 것은 문법이 아니라 흐름이다. 파일을 열었을 때 무엇을 준비하고, 무엇을 판단하고, 무엇을 반환하는지가 자연스럽게 이어져야 한다.

## 이름은 작은 문서다

변수와 함수 이름은 가장 자주 읽히는 문서다. 주석보다 가까이 있고, 테스트보다 먼저 눈에 들어온다. 그래서 이름에는 구현 방법보다 역할이 드러나는 편이 좋다.

```js
const expiredSubscriptions = subscriptions.filter((subscription) => {
    return subscription.expiresAt < today;
});
```

위 코드에서 중요한 것은 필터 조건 자체보다 `expiredSubscriptions`라는 결과의 의미다. 나중에 이 값을 넘겨받는 코드는 조건식을 다시 읽지 않아도 맥락을 이해할 수 있다.

## 함수는 한 번에 하나의 질문만 답한다

함수가 여러 질문에 동시에 답하기 시작하면 수정하기 어려워진다. 예를 들어 권한을 확인하고, 데이터를 불러오고, 응답 모양까지 만드는 함수는 재사용하기도 테스트하기도 어렵다.

좋은 기준은 다음 세 가지다.

1. 함수 이름을 읽고 결과를 예상할 수 있는가
2. 조건문이 너무 깊어지지 않는가
3. 실패했을 때 어느 단계가 문제인지 바로 보이는가

이 기준을 통과하지 못한다면 함수를 나누는 편이 낫다.

## 기록은 코드 옆에 짧게 둔다

모든 판단을 긴 문서로 남길 필요는 없다. 대신 나중에 헷갈릴 만한 선택은 코드 가까이에 짧게 남겨두면 좋다.

```js
// 외부 API가 초 단위 timestamp를 반환하므로 내부 기준인 millisecond로 맞춘다.
const expiresAt = response.expires_at * 1000;
```

이런 주석은 코드가 무엇을 하는지 반복하지 않는다. 왜 그렇게 했는지를 알려준다. 시간이 지난 뒤에도 같은 결정을 반복해서 추적하지 않게 해준다.

## 나중에 고칠 사람을 위한 체크리스트

- 이름은 결과와 역할을 말한다.
- 함수는 한 가지 판단만 담당한다.
- 조건문은 읽는 순서대로 배치한다.
- 주석은 이유를 남길 때만 쓴다.
- 예외 상황은 조용히 숨기지 않는다.

코드는 늘 바뀐다. 그래서 완벽한 구조보다 다시 고칠 수 있는 구조가 더 오래 간다.
